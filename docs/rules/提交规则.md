# 提交规则

## 核心原则

### 1. 默认行为：多个提交（不可妥协）

**单个提交 = 自动失败**

默认行为是创建多个提交。

| 文件变更数 | 最少提交数 |
|------------|------------|
| 3 个文件   | 2 次提交   |
| 5 个文件   | 2 次提交   |
| 9 个文件   | 3 次提交   |
| 15 个文件  | 5 次提交   |

**硬性规则：** 如果从 3 个以上的文件只做 1 次提交，你错了。停止并拆分。

### 2. 拆分标准

| 标准 | 操作 |
|------|------|
| 不同目录/模块 | 拆分 |
| 不同组件类型（model/service/view） | 拆分 |
| 可以独立回滚 | 拆分 |
| 不同关注点（UI/逻辑/配置/测试） | 拆分 |
| 新文件 vs 修改 | 拆分 |

### 3. 仅在以下全部满足时合并

- 确实是相同的原子单元（如：函数 + 其测试文件）
- 拆分会破坏编译
- 能用一句话说明原因

## 提交信息风格

### 语言检测

从最近的提交统计：
- 韩文字符 ≥ 50% → 韩文
- 英文 ≥ 50% → 英文
- 混合 → 使用多数语言

### 风格分类

| 风格 | 模式 | 示例 |
|------|------|------|
| `SEMANTIC` | `type: message` 或 `type(scope): message` | `feat: add login` |
| `PLAIN` | 纯描述，无前缀 | `Add login feature` |
| `SENTENCE` | 完整语法句子 | `Implemented the new login flow` |
| `SHORT` | 最小化关键词 | `format`, `lint` |

### 风格检测算法

```
semantic_count = 匹配语义正则的提交数
plain_count = 非语义提交中 >3 个词的提交数
short_count = ≤3 个词的提交数

IF semantic_count >= 15 (50%): STYLE = SEMANTIC
ELSE IF plain_count >= 15: STYLE = PLAIN  
ELSE IF short_count >= 10: STYLE = SHORT
ELSE: STYLE = PLAIN (安全默认值)
```

## 分支规则

### 分支状态评估

```
IF current_branch == main OR current_branch == master:
  -> 禁止重写历史

ELSE IF commits_ahead == 0:
  -> 仅安全新增提交

ELSE IF 所有提交都是本地（未推送）:
  -> 可进行激进重写

ELSE IF 已推送但未合并:
  -> 谨慎重写，警告 force push
```

### 历史重写安全性

| 条件 | 风险 | 操作 |
|------|------|------|
| 在 main/master 上 | 危险 | 禁止重写 |
| 工作区不干净 | 警告 | 先 stash |
| 存在已推送提交 | 警告 | 确认后再 force push |
| 所有提交都是本地 | 安全 | 可自由操作 |

## 原子提交规划

### 最少提交数公式

```
min_commits = ceil(file_count / 3)
```

### 必要理由说明

对于每个包含 3+ 个文件的提交，必须说明：

```
"提交 N 包含 [文件列表]，因为 [它们不可分割的具体原因]。"
```

**有效理由：**
- "实现文件 + 其直接测试文件"
- "类型定义 + 使用它的唯一文件"
- "迁移 + 模型变更（没有两者会出错）"

**无效理由（必须拆分）：**
- "都与功能 X 相关"（太模糊）
- "同一 PR 的一部分"（不是理由）
- "它们一起修改了"（不是理由）

### 依赖顺序

```
Level 0: 工具类、常量、类型定义
Level 1: 模型、schema、接口
Level 2: 服务、业务逻辑
Level 3: API 端点、控制器
Level 4: 配置、基础设施

提交顺序: Level 0 -> Level 1 -> Level 2 -> Level 3 -> Level 4
```

### 测试与实现配对

```
规则: 测试文件必须与实现文件在同一提交中

测试模式:
- test_*.py <-> *.py
- *_test.py <-> *.py
- *.test.ts <-> *.ts
- *.spec.ts <-> *.ts
- __tests__/*.ts <-> *.ts
```

## 提交执行

### 执行流程

1. 为每个逻辑单元暂存文件
2. 使用 `git diff --staged --stat` 验证暂存
3. 使用检测到的风格提交
4. 使用 `git log -1 --oneline` 验证

### 共同作者 footer（必须）

每个提交必须包含：

```
Ultraworked with [Sisyphus](https://github.com/code-yeongyu/oh-my-opencode)

Co-authored-by: Sisyphus <clio-agent@sisyphuslabs.ai>
```

### 提交命令模板

```bash
git commit -m "{message}" -m "Ultraworked with [Sisyphus](https://github.com/code-yeongyu/oh-my-opencode)" -m "Co-authored-by: Sisyphus <clio-agent@sisyphuslabs.ai>"
```

## 反模式（自动失败）

1. ❌ **禁止单个大提交** - 3+ 文件必须 2+ 次提交
2. ❌ **不要默认语义提交** - 先从 git log 检测
3. ❌ **禁止分离测试与实现** - 必须在一起
4. ❌ **不要按文件类型分组** - 按功能/模块分组
5. ❌ **未经明确许可禁止重写已推送历史**
6. ❌ **禁止留下未完成的工作区** - 完成所有变更
7. ❌ **禁止跳过理由说明** - 解释为什么文件要分组
8. ❌ **禁止模糊的分组理由** - "与 X 相关" 不是理由

## 验证清单

创建任何提交前：

- [ ] 文件数量检查: N 个文件 → 至少 ceil(N/3) 次提交？
- [ ] 理由检查: 每个 3+ 文件的提交是否说明了原因？
- [ ] 目录拆分检查: 不同目录 → 不同提交？
- [ ] 测试配对检查: 每个测试与其实现配对？
- [ ] 依赖顺序检查: 基础优先于依赖项？

---

## 硬性停止条件

- 从 3+ 个文件做 1 次提交 → **错误。拆分。**
- 从 10+ 个文件做 2 次提交 → **错误。继续拆分。**
- 无法用一句话说明文件分组理由 → **错误。拆分。**
- 不同目录在同一次提交（无理由）→ **错误。拆分。**
