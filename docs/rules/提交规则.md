# 提交规则

## 核心原则

### 1. 默认行为：多个提交（不可妥协）

**单个提交 = 自动失败**

默认行为是创建多个提交。

| 文件变更数 | 最少提交数 |
|------------|------------|
| 3 个文件   | 2 次提交   |
| 5 个文件   | 2 次提交   |
| 9 个文件   | 3 次提交   |
| 15 个文件  | 5 次提交   |

**硬性规则：** 如果从 3 个以上的文件只做 1 次提交，你错了。停止并拆分。

### 2. 拆分标准

| 标准 | 操作 |
|------|------|
| 不同目录/模块 | 拆分 |
| 不同组件类型（model/service/view） | 拆分 |
| 可以独立回滚 | 拆分 |
| 不同关注点（UI/逻辑/配置/测试） | 拆分 |
| 新文件 vs 修改 | 拆分 |

### 3. 仅在以下全部满足时合并

- 确实是相同的原子单元（如：函数 + 其测试文件）
- 拆分会破坏编译
- 能用一句话说明原因

### 4. 仅提交暂存区的代码

**禁止使用 `git commit -a` 或 `git commit --all`，禁止提交未显式暂存的文件。**

每次提交前必须：
1. 明确暂存目标文件：`git add <file>`
2. 用 `git diff --staged --stat` 确认暂存内容
3. 确保工作区中不存在未暂存的相关变更

**原因**：未暂存的变更可能在不知情的情况下被遗漏，导致提交不完整或引入无关修改。

## 提交信息风格

本项目采用 **约定式提交（Conventional Commits）** 作为语义化提交的标准格式，
参考 [Conventional Commits v1.0.0](https://www.conventionalcommits.org/zh-hans/v1.0.0/)。

### 1. 提交信息结构

```
<类型>[可选 范围]: <描述>

[可选 正文]

[可选 脚注]
```

### 2. 类型定义

| 类型 | 含义 | 语义版本对应 |
|------|------|------------|
| `feat` | 新增功能 | MINOR |
| `fix` | 修复 bug | PATCH |
| `docs` | 文档变更 | — |
| `style` | 代码格式（不影响功能） | — |
| `refactor` | 重构（非功能修改） | — |
| `perf` | 性能优化 | — |
| `test` | 测试相关 | — |
| `build` | 构建系统/依赖变更 | — |
| `chore` | 非业务代码修改 | — |
| `ci` | 持续集成配置 | — |
| `revert` | 回退提交 | — |

### 3. 范围（Scope）

范围附在类型后的圆括号中，描述代码的某部分：

```
feat(parser): 新增数组解析能力
fix(api): 修复认证超时问题
docs(readme): 更新安装说明
```

### 4. 破坏性变更（BREAKING CHANGE）

破坏性变更使用 `!` 标记，或在脚注中写明：

```
# 方式一：使用 !
feat!: 移除旧版 API
feat(api)!: 废弃 v1 接口

# 方式二：脚注
feat: 支持配置继承

BREAKING CHANGE: extends 键现在用于继承其他配置文件
```

- `!` 标记 + 脚注 `BREAKING CHANGE:` 都存在时，只需一种即可
- 破坏性变更对应语义版本 **MAJOR**

### 5. 正文与脚注规范

**正文**：描述字段后空一行开始，可分段，可空行分隔段落。

**脚注**：令牌 + `: ` + 值，参考 [git trailer format](https://git-scm.com/docs/git-interpret-trailers)：

```
fix: 防止请求竞态

引入请求 ID 并引用最新请求。
忽略来自非最新请求的响应。

Reviewed-by: Zhang San
Refs: #123
```

### 6. 提交信息规则（RFC 2119 关键词摘要）

1. 每个提交 **必须** 使用 `<类型>:` 前缀
2. 新增功能 **必须** 使用 `feat`
3. 修复 bug **必须** 使用 `fix`
4. 范围 **可以** 跟在类型后，用圆括号包裹
5. 描述字段 **必须** 直接跟在类型前缀的冒号空格后
6. 正文 **可以** 在描述后空一行开始
7. 脚注 **可以** 在正文后空一行开始
8. 破坏性变更 **必须** 标记（`!` 或脚注 `BREAKING CHANGE:`）
9. 工具 **必须** 不区分大小写解析（`BREAKING CHANGE` 除外，必须大写）

### 7. 与本项目多提交规则的关系

- **本项目的多提交、原子拆分规则优先于提交信息格式**
- 语义化类型（feat/fix/docs…）用于每一条提交的**前缀格式**
- Scope 范围进一步说明影响模块
- 两条规则共同生效：先确保拆分正确，再确保格式合规

---

### 强制风格：Conventional Commits

本项目**强制要求**使用 Conventional Commits 格式（`SEMANTIC`），不得降级为 PLAIN / SENTENCE / SHORT。

### 语言检测

提交信息语言由以下规则决定：

| 条件 | 语言 |
|------|------|
| 最近提交中韩文字符 ≥ 50% | 韩文 |
| 最近提交中英文字符 ≥ 50% | 英文 |
| 混合 | 使用多数语言 |

**描述（description）语言**：跟随语言检测结果。
**类型（type）**：英文小写，不区分大小写解析（`BREAKING CHANGE` 除外，必须大写）。

## 分支规则

### 分支状态评估

```
IF current_branch == main OR current_branch == master:
  -> 禁止重写历史

ELSE IF commits_ahead == 0:
  -> 仅安全新增提交

ELSE IF 所有提交都是本地（未推送）:
  -> 可进行激进重写

ELSE IF 已推送但未合并:
  -> 谨慎重写，警告 force push
```

### 历史重写安全性

| 条件 | 风险 | 操作 |
|------|------|------|
| 在 main/master 上 | 危险 | 禁止重写 |
| 工作区不干净 | 警告 | 先 stash |
| 存在已推送提交 | 警告 | 确认后再 force push |
| 所有提交都是本地 | 安全 | 可自由操作 |

## 原子提交规划

### 最少提交数公式

```
min_commits = ceil(file_count / 3)
```

### 必要理由说明

对于每个包含 3+ 个文件的提交，必须说明：

```
"提交 N 包含 [文件列表]，因为 [它们不可分割的具体原因]。"
```

**有效理由：**
- "实现文件 + 其直接测试文件"
- "类型定义 + 使用它的唯一文件"
- "迁移 + 模型变更（没有两者会出错）"

**无效理由（必须拆分）：**
- "都与功能 X 相关"（太模糊）
- "同一 PR 的一部分"（不是理由）
- "它们一起修改了"（不是理由）

### 依赖顺序

```
Level 0: 工具类、常量、类型定义
Level 1: 模型、schema、接口
Level 2: 服务、业务逻辑
Level 3: API 端点、控制器
Level 4: 配置、基础设施

提交顺序: Level 0 -> Level 1 -> Level 2 -> Level 3 -> Level 4
```

### 测试与实现配对

```
规则: 测试文件必须与实现文件在同一提交中

测试模式:
- test_*.py <-> *.py
- *_test.py <-> *.py
- *.test.ts <-> *.ts
- *.spec.ts <-> *.ts
- __tests__/*.ts <-> *.ts
```

## 提交执行

### 执行流程

1. 为每个逻辑单元暂存文件
2. 使用 `git diff --staged --stat` 验证暂存
3. 使用检测到的风格提交
4. 使用 `git log -1 --oneline` 验证

### 共同作者 footer（必须）

每个提交必须包含：

```
Ultraworked with [Sisyphus](https://github.com/code-yeongyu/oh-my-opencode)

Co-authored-by: Sisyphus <clio-agent@sisyphuslabs.ai>
```

### 提交命令模板

```bash
git commit -m "{message}" -m "Ultraworked with [Sisyphus](https://github.com/code-yeongyu/oh-my-opencode)" -m "Co-authored-by: Sisyphus <clio-agent@sisyphuslabs.ai>"
```

## 反模式（自动失败）

1. ❌ **禁止单个大提交** - 3+ 文件必须 2+ 次提交
2. ❌ **禁止非 Conventional Commits 格式** - 必须使用 `type: description` 或 `type(scope): description`
3. ❌ **禁止分离测试与实现** - 必须在一起
4. ❌ **不要按文件类型分组** - 按功能/模块分组
5. ❌ **未经明确许可禁止重写已推送历史**
6. ❌ **禁止留下未完成的工作区** - 完成所有变更
7. ❌ **禁止跳过理由说明** - 解释为什么文件要分组
8. ❌ **禁止模糊的分组理由** - "与 X 相关" 不是理由
9. ❌ **禁止提交未暂存的文件** - 必须 `git add` 显式暂存后再提交

## 验证清单

创建任何提交前：

- [ ] **暂存检查**：是否只暂存了目标文件（无意外文件，无遗漏文件）？
- [ ] **格式检查**：是否符合 `<type>[:!] <description>` 或 `<type(scope)>[:!] <description>` 格式？
- [ ] 类型检查：类型是否为合法关键字（feat/fix/docs/style/refactor/perf/test/build/chore/ci/revert）？
- [ ] 破坏性变更检查：如有破坏性变更，是否标记了 `!` 或脚注 `BREAKING CHANGE:`？
- [ ] 文件数量检查: N 个文件 → 至少 ceil(N/3) 次提交？
- [ ] 理由检查: 每个 3+ 文件的提交是否说明了原因？
- [ ] 目录拆分检查: 不同目录 → 不同提交？
- [ ] 测试配对检查: 每个测试与其实现配对？
- [ ] 依赖顺序检查: 基础优先于依赖项？

---

## 硬性停止条件

- 从 3+ 个文件做 1 次提交 → **错误。拆分。**
- 从 10+ 个文件做 2 次提交 → **错误。继续拆分。**
- 无法用一句话说明文件分组理由 → **错误。拆分。**
- 不同目录在同一次提交（无理由）→ **错误。拆分。**
